# 个人发票与消费分析助手 (Personal Invoice Assistant) - 终版产品需求文档 (PRD)

## 1. 产品概述与核心价值
* **产品定位：** 一款全云端部署、重隐私保护、具备极致现代化视觉体验的个人发票管理、报销追踪与财务分析系统。
* **核心痛点：** 解决个人差旅、日常消费中发票归集碎片化、手动录入低效、重复报销难拦截、消费数据无法结构化分析、报销状态流转无记录的痛点。
* **核心技术栈红线（开发 Agent 必须严格遵守）：**
    * **前端 (Frontend)：** React + Vite + TailwindCSS + **Horizon UI PRO 模板规范** + Tremor (数据看板图表库)。
    * **后端 (Backend)：** Python + FastAPI (全异步架构) + PostgreSQL (关系型数据库)。
    * **对象存储 (Storage)：** 任何 S3 兼容的云端对象存储 (如 Cloudflare R2 / AWS S3)，严禁在容器本地持久化存储发票原文件。
    * **部署 (Deployment)：** 纯 Docker 容器化编排，提供完整的 `Dockerfile` 与 `docker-compose.yml`。

## 3. 功能需求详细说明

### 3.1 模块一：工作台与多源文件归集 (Dashboard & Ingestion)
* **3.1.1 拖拽上传组件 (Dropzone)：**
    * **页面位置：** `/dashboard` 工作台页面顶部。
    * **交互表现：** 现代极简虚线边框，支持拖拽与点击上传。支持格式：`.pdf`, `.png`, `.jpg`, `.jpeg`。支持多文件批量拖拽，进入后端异步排队解析队列。
    * **状态反馈：** 上传后列表即时生成记录，状态 Badge 显示为闪烁的 `识别中(Processing)...`。
* **3.1.2 QQ 邮箱 IMAP 自动归集后台 (Email IMAP Worker)：**
    * **运行机制：** 后端挂载一个独立的异步定时任务 Worker（建议基于 `aioimaplib`，每 15-30 分钟轮询一次）。
    * **连接配置：** * IMAP 服务器: `imap.qq.com` | 端口: `993` (必须启用 SSL)。
        * 认证凭证: 用户 QQ 邮箱号 + **16位第三方应用专属授权码**（从 QQ 邮箱网页端“设置-账户”中生成，严禁使用 QQ 登录密码）。
    * **过滤与拦截：**
        * 状态过滤：仅检索收件箱中 `UNSEEN`（未读）状态的邮件。
        * 强关键词匹配：邮件主题或正文必须包含：`发票`、`电子发票`、`行程单`、`Invoice` 之一，否则静默忽略。
    * **解析兼容性红线：**
        1. 优先提取符合格式要求的 `.pdf`、`.png`、`.jpg` 邮件附件。
        2. **特殊邮件兼容：** 针对国内部分服务商将发票以 Inline Base64 图片或外部图片 URL 直接嵌入 HTML 正文而无附件的情况，Worker 必须具备解析邮件正文 HTML 的能力，抓取该图片文件流，同样送入 AI 提取队列。

### 3.2 模块二：智能提取与沉浸式校对大表 (Invoices Management)
* **3.2.1 数据大表 (Complex Table)：**
    * **页面位置：** `/invoices` 页面。
    * **UI 规范：** 必须百分之百复用 **Horizon UI PRO 的 `ComplexTable`** 设计，采用高级黑白灰与浅色/深色模式响应。
    * **列定义：** 开票日期、发票类型、开票方、价税合计、归属分类、数据来源（`手动上传`/`QQ邮箱`）、报销状态（🔴 `待报销` / 🟡 `报销中` / 🟢 `已到账`）。
    * **快捷操作：** 表格顶部常驻 Tab 过滤器：`全部` | `待报销` | `报销中` | `已完成`。操作列支持不进入详情直接右键或点击修改报销状态。
* **3.2.2 沉浸式校对抽屉 (Slide-over Drawer)：**
    * **交互触发：** 点击表格中任意 `待校对` 记录，从右侧滑出抽屉（宽度占屏幕 50%）。
    * **左侧分栏：** 发票原件高保真预览区。支持 PDF 渲染或图片渲染，支持鼠标滚轮或双击局部放大（方便查看细小校验码）。
    * **右侧分栏：** 结构化可编辑表单（字段见数据字典）。
* **3.2.3 智能防重校验引擎 (Anti-Duplicate)：**
    * 在抽屉表单初始化、用户手动输入或最终点击保存时，后端必须执行 `发票代码 (invoice_code) + 发票号码 (invoice_number)` 联合唯一性检索。
    * 若在数据库中检索到已存在记录，前端对应字段爆红高亮，并弹出强警告提示：“该发票已于 YYYY-MM-DD 录入系统，存在重复报销风险！”，同时置灰“确认入库”按钮。

### 3.3 模块三：消费分析与财务看板 (Consumption Analytics)
* **3.3.1 AI 自动分类打标 (Smart Categorization)：**
    * 后端调用多模态大模型时，通过 Prompt 引导，要求大模型根据开票方名称及发票明细进行智能分类（如：`AWS/阿里云` -> `数字服务`；`滴滴出行/高铁` -> `差旅出行`；`Shimano` -> `运动爱好`）。
    * 支持用户在前端手动纠正分类，系统需在后端记录此“开票方-分类”映射规则，后续同类开票方自动应用该修正规则。
* **3.3.2 看板视觉呈现 (Dashboard)：**
    * **页面位置：** `/analytics` 页面。完全复用 **Horizon UI PRO 的 `Main Dashboard` 布局**。
    * **指标卡 (MiniStatistics)：** 顶部并排展示三个高精美卡片：区间总消费、**待报销总额**、最大单笔支出。点击“待报销总额”卡片可自动跳转至列表并应用“待报销”过滤。
    * **数据图表（使用 Tremor 组件）：**
        * 消费分类占比环形图 (Donut Chart)：直观展示资金开销结构，支持点击环形块向下钻取明细。
        * 月度消费趋势面积图 (Area Chart)：展示开销的时间演变趋势。

### 3.4 模块四：数据流转与打包导出 (Export & Workflow)
* **3.4.1 导出即提交流转逻辑：**
    * 用户在列表中勾选多张发票（通常为多张 `待报销` 状态票据），点击“导出报销单”。
    * **系统交互确认：** 前端弹出精美 Modal 确认框：“报销单 Excel 已生成。是否同步将选中的 X 张发票状态变更为「报销中 (Submitted)」？”
    * **状态联动：** 用户确认后，后端生成 ZIP 文件的同时，批量异步更新数据库中这批记录的 `reimbursement_status = 'submitted'`。
    * **最终产出物：** 一个标准 `.zip` 压缩包，内含：
        1. `invoices_export.xlsx`（包含选定发票所有结构化字段的对账单）。
        2. 按 `开票日期_开票方_金额` 规范自动重命名的发票原件文件夹。

## 4. 技术架构与数据库设计 (开发边界)

### 4.1 核心数据库字典 (Database Schema)

#### 表名：`invoices`
| 字段名 | 类型 | 约束 | 描述 |
| :--- | :--- | :--- | :--- |
| `id` | UUID | Primary Key | 唯一标识 |
| `invoice_code` | VARCHAR(64) | 可为空 | 发票代码 (新版全电发票为空) |
| `invoice_number` | VARCHAR(64) | NOT NULL | 发票号码 (核心去重索引组合) |
| `issue_date` | DATE | NOT NULL | 开票日期 |
| `invoice_type` | VARCHAR(32) | NOT NULL | 枚举: 专票/普票/全电/打车票/其他 |
| `seller_name` | VARCHAR(255) | NOT NULL | 销售方 / 开票单位名称 |
| `buyer_name` | VARCHAR(255) | NOT NULL | 购买方抬头 |
| `total_amount` | NUMERIC(12, 2)| NOT NULL | 价税合计总金额 (浮点数) |
| `category` | VARCHAR(64) | NOT NULL | 账目分类 (如: 数字服务/差旅出行/日常餐饮) |

| `source` | VARCHAR(32) | NOT NULL | 渠道来源 (枚举: `manual`, `email_auto`) |
| `status` | VARCHAR(32) | NOT NULL | AI校对状态 (枚举: `pending`, `verified`) |

| 字段名 | 类型 | 约束 | 描述 |
| :--- | :--- | :--- | :--- |
| `log_id` | UUID | Primary Key | 日志唯一ID |
| `sync_time` | TIMESTAMP | NOT NULL | 同步触发时间 |
| `sender` | VARCHAR(255) | NOT NULL | 发件人邮箱 |

| `error_message` | TEXT | NULL | 失败时的堆栈异常简述 |

### 4.2 数据主权与隐私安全控制红线
1. **存储沙箱化：** 存储桶（S3/R2）必须设置为 **完全私有 (Absolute Private)**。绝对不允许在前端暴露任何带有 S3 公网域名的直连 URL。
2. **安全预览层：** 前端在渲染发票原件预览时，必须向后端发起请求。由 FastAPI 后端通过 `boto3` SDK 生成一个**时效仅为 60 秒的预签名 URL (Presigned URL)** 返回给前端，前端使用该临时 URL 进行渲染，到期自动失效。
3. **凭证隐蔽：** IMAP 绑定的 QQ 邮箱账户名与 16 位传输加密授权码必须通过服务端的本地环境文件 (`.env`) 传入，严禁在任何前端页面以明文表单或本地缓存形式留存。
4. **解耦大模型：** 后端 AI 解析层必须设计为抽象 Client。其请求的 Base URL 应指向用户内部的 AIGC 聚合网关，严禁将原生的 OpenAI/Anthropic 等第三方 SDK 强行绑定在发票业务逻辑中。

## 5. Agent 开发提示词流水线 (Prompting Pipeline)

在您本地配置好基础工程环境后，请**按照以下前后顺序**分步骤向您的编码 Agent (如 Cursor) 喂入指令：

### 阶段一：前端脚手架瘦身与基础路由建立
> "我们现在克隆了 Horizon UI PRO 的 Tailwind React 模板。请首先全局阅读项目目录。为了使项目保持干净，请彻底清理掉 `src/views/` 下与 NFT 市场、Kanban、Calendar 等无关的全部业务 Demo。重构路由配置，按照 PRD 第 3 章节建立 `/dashboard`, `/invoices`, `/analytics` 三个空白路由页，并使用优雅的图标更新 Sidebar 菜单。确保原模板的暗黑模式和全局 Layout 完美保留。"

### 阶段二：开发发票大表与沉浸式状态校对抽屉
> "请在 `/invoices` 页面中，深度复用模板原生的 `ComplexTable` 组件来渲染发票列表。参考 PRD 3.2 章节，增加报销状态列，并使用带有鲜明颜色区分的 Badge（待报销/报销中/已到账）进行渲染。表格顶部增加快捷过滤器 Tabs。接下来，使用 Headless UI 或组件库实现右侧滑出的校对抽屉（50%宽度），抽屉内部使用 Tailwind 的 `grid-cols-2` 进行左右分栏：左侧预留图片位，右侧渲染结构化可编辑表单。请先使用硬编码的 Mock JSON 数据完成全部前端交互逻辑设计。"

### 阶段三：搭建 FastAPI 后端、数据层与 QQ 邮箱 IMAP Worker
> "请基于 Python FastAPI 构建系统后端。根据 PRD 4.1 章节的数据字典，编写 SQLAlchemy 数据库模型与 Pydantic 验证模型（包含 invoices 表与 email_sync_logs 表）。利用 Python 内置的 `imaplib` 或 `aioimaplib` 编写一个后台异步定时轮询 Worker。Worker 需要连接环境变量中配置的 QQ 邮箱（imap.qq.com, 993 端口），能够过滤未读邮件，并深度解析 MIME 结构（必须包含解析邮件正文 HTML 提取内嵌 Base64 图片/链接的能力）。提取的文件和手动上传的文件，统一通过 `boto3` 标准库写入 S3 兼容的对象存储，并实现时效 60 秒的预签名预览安全机制。"

### 阶段四：集成 AI 解析网关与导出流转闭环
> "最后，编写发票的智能解析与归类逻辑。接收后端暂存的文件流，将其投递给环境变量中配置的 AIGC 聚合网关。提供严谨的 System Prompt，要求多模态大模型提取全字段，并根据开票方自动返回对应的 `category` 和严格的 JSON 结构。随后，实现批量导出接口：接收前端传来的发票 ID 列表，使用 `pandas` 或 `openpyxl` 生成对账 Excel，将发票原件重命名后，统一打包成 ZIP 返回下载流。在下载触发时，异步将这批记录的 `reimbursement_status` 批量原子化更新为 `submitted`（报销中）。"
